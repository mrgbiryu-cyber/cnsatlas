# PPT to Figma 변환 품질 검증 초안

## 1. 목적

이 문서는 PPT to Figma 변환 품질을 어떻게 검증할지 정의한다.

이 프로젝트에서 품질 검증의 목적은 단순히 "PPT와 비슷하게 보이는가"를 확인하는 것이 아니다.

반드시 아래 3가지를 동시에 검증해야 한다.

1. PPT를 구조적으로 잘 해석했는가
2. Figma에서 살아있는 편집 구조로 변환되었는가
3. 변환 결과가 이후 sync, 검색, 정책 연결, 담당 연결에 재사용 가능한가

즉, 품질 검증은 시각 품질 + 구조 품질 + 운영 활용 가능성을 모두 확인해야 한다.

## 2. 품질 검증의 기본 원칙

### 2-1. 사람에게 정밀 채점을 맡기지 않는다

사람이 모든 변환 결과를 세밀하게 점수화하는 방식은 비현실적이다.

이유:

- 문서 수가 많아지면 검토 비용이 급증한다
- 검토자마다 기준이 달라진다
- "보기 비슷함"과 "실제 작업 가능함"은 다르다

따라서 검증은 아래 2층으로 나눠야 한다.

1. 자동검증
- 구조적 최소 합격선 판정
- 위험 문서 자동 분류

2. 사람검증
- 실제 기획자가 그 결과물로 작업을 이어갈 수 있는지 판정

### 2-2. 육안 검토만으로 끝내지 않는다

사람이 보기 좋은 결과만으로 합격시키면 안 된다.

예:

- 화면은 비슷하지만 레이어가 전부 납작해진 경우
- 표가 이미지처럼 깨진 경우
- 텍스트가 편집 불가능한 경우
- 섹션/셀/영역 단위 노드가 사라진 경우

이런 결과는 운영 플랫폼 관점에서는 실패다.

### 2-3. 품질은 3계층으로 본다

품질은 아래 3계층으로 평가한다.

1. 시각 품질
2. 구조 품질
3. 운영 적합성

이 3계층 중 하나라도 심하게 무너지면 고품질 변환으로 보지 않는다.

### 2-4. 샘플 기반 검증이 필요하다

품질은 임의 문서 1개로 판단하면 안 된다.

반드시 서로 다른 유형의 PPT 샘플 세트를 만들고, 유형별로 반복 검증해야 한다.

현재 benchmark set v1은 아래와 같이 고정한다.

- 파일: `sampling/pptsample.pptx`
- 전체 슬라이드 수: 38
- 핵심 기준 슬라이드: `12`, `26`, `29`, `34`

이 4개 슬라이드는 1차 변환 품질 검증에서 필수 우선 검토 대상으로 사용한다.

## 3. 품질 검증 프레임

### 3-1. 검증 책임 분리

권장 분리:

- 자동검증: 엔진이 `pass`, `warn`, `fail` 판정
- 사람검증: 기획자가 `usable`, `usable_with_fix`, `not_usable` 판정

자동검증은 구조적 건강도를 본다.

사람검증은 실무 지속 가능성을 본다.

### 3-2. 1차 평가 축

#### A. 시각 품질

질문:

- 원본 PPT와 Figma 결과가 충분히 유사한가
- 레이아웃 붕괴가 없는가
- 텍스트 위치, 크기, 정렬이 크게 깨지지 않았는가
- 색상, 선, 도형, 간격이 지나치게 무너지지 않았는가

#### B. 구조 품질

질문:

- slide, group, section, table, row, cell, text block 구조가 살아 있는가
- Figma에서 의미 있는 frame/group/node 계층으로 남았는가
- 반복되는 UI가 컴포넌트 후보로 식별 가능한가
- 텍스트가 editable 상태인가
- 표/셀을 node 수준으로 추적 가능한가

#### C. 운영 적합성

질문:

- 변환된 node에 annotation을 붙일 수 있는가
- 특정 영역에 ownership을 연결할 수 있는가
- source_mapping이 안정적으로 생성되는가
- 이후 Figma incremental pull 시 identity를 유지할 가능성이 있는가
- node/page/document 단위 검색 projection이 가능한가

## 4. 세부 평가 항목

### 4-1. 시각 품질 항목

- 레이아웃 유사도
- 텍스트 위치 유사도
- 텍스트 스타일 보존율
- 색상/도형 속성 보존율
- 표 시각 구조 보존율
- 이미지/아이콘 위치 보존율
- 겹침/잘림/오버플로 발생 여부

### 4-2. 구조 품질 항목

- slide -> page 변환 정확도
- group/frame 계층 보존율
- section/영역 분리 정확도
- table 인식 정확도
- row/cell 인식 정확도
- text block 독립성
- component 후보 식별 가능성
- editable text 유지율
- asset 분리 정확도

### 4-3. 운영 적합성 항목

- node 단위 source_mapping 생성율
- stable identity 생성 가능성
- annotation target 적합성
- ownership target 적합성
- policy link target 적합성
- 검색 인덱싱 가능률
- diff 비교 가능성
- remap 가능성

## 5. 자동검증과 사람검증의 역할 분리

### 5-1. 자동검증이 담당할 것

자동검증은 아래를 본다.

- 최소 구조가 살아 있는가
- 운영 기능에 필요한 핵심 데이터가 남아 있는가
- 위험 문서로 분류해야 하는가

자동검증의 목적은 완벽함 증명이 아니라, 운영 불가 수준의 실패를 빠르게 걸러내는 것이다.

### 5-2. 사람검증이 담당할 것

사람검증의 최종 질문은 하나다.

"이 변환 결과물로 Figma에서 바로 기획 작업을 이어갈 수 있는가?"

사람은 아래를 본다.

- 텍스트를 바로 수정할 수 있는가
- 표, 셀, 영역 수정이 가능한가
- 레이어 구조가 이해 가능한가
- 반복 작업이나 복제가 가능한가
- 일부 보정만 하면 계속 사용할 수 있는가

즉 사람검증은 미관 심사가 아니라 업무 사용 가능성 판정이다.

### 5-3. 최종 판정 방식

권장 조합:

- `auto_pass + usable`: 바로 운영 가능
- `auto_pass + usable_with_fix`: 반자동 보정 후 사용
- `auto_warn + usable`: 제한적 사용 가능
- `auto_fail`: 운영 투입 금지

## 6. 평가 등급 체계

권장 등급:

- A: 운영 즉시 가능
- B: 소규모 수동 보정 후 운영 가능
- C: 시각은 유사하나 구조 보정 필요
- D: 운영용으로 부적합

### A 등급 기준 예시

- 핵심 레이아웃 보존
- 텍스트 editable
- 표/셀 구조 대부분 유지
- node 계층이 운영 연결에 충분
- source_mapping 안정적

### B 등급 기준 예시

- 전체 구조는 유지되나 일부 그룹/셀 보정 필요
- 운영에는 사용 가능하나 검수 작업 필요

### C 등급 기준 예시

- 겉보기는 비슷하지만 node 구조가 약해 검색/sync 연결이 어려움

### D 등급 기준 예시

- 이미지 수준 변환
- 계층 붕괴
- 텍스트 수정 불가
- 운영 연결 불가

## 7. 샘플 세트 설계

품질 검증용 샘플 PPT는 아래 유형을 반드시 포함해야 한다.

현재는 `sampling/pptsample.pptx`를 공식 benchmark set v1로 사용한다.

운영 기준:

- 전체 38슬라이드는 회귀 테스트 대상이다.
- 슬라이드 `12`, `26`, `29`, `34`는 필수 우선 검증 슬라이드다.
- 자동검증 리포트와 사람검수 리포트는 최소한 위 4개 슬라이드 기준 결과를 반드시 분리 표시해야 한다.

### 7-1. 기본 화면형

- 단순 텍스트 + 버튼 + 박스 구조
- 일반적인 화면 설계서

목적:

- 기본 text/shape/frame 변환 검증

### 7-2. 표 중심형

- 표, 행, 열, 셀이 많은 문서
- 셀 병합이 포함된 문서

목적:

- table/row/cell 구조 보존 검증

### 7-3. 그룹/영역 복잡형

- 그룹, 섹션, 묶음이 많은 문서
- nested group이 있는 문서

목적:

- 계층 보존 검증

### 7-4. 이미지/아이콘 혼합형

- 이미지, 아이콘, 배경, 썸네일이 많은 문서

목적:

- asset 분리 및 위치 보존 검증

### 7-5. 반복 UI 패턴형

- 카드 목록, 반복 폼, 리스트 구조가 많은 문서

목적:

- component 후보 식별 가능성 검증

### 7-6. 실제 난이도 높은 현업 문서형

- 실제 업무에서 쓰는 난이도 높은 PPT
- 정렬 불균일, 수동 배치, 복잡한 표, 섞인 주석 포함

목적:

- 실전 적합성 검증

## 8. 검증 절차

### 8-1. 1단계: 자동 구조 검증

확인할 것:

- slide 수가 정확한가
- 텍스트 박스 수가 크게 누락되지 않았는가
- 표와 셀을 분리 인식하는가
- 그룹과 섹션 후보를 추출하는가
- source path를 남기는가

결과물:

- parser 결과 리포트
- confidence 리포트
- 누락/오탐 목록

자동검증에서 관리할 후보 지표:

- 페이지 매칭률
- 텍스트 추출 성공률
- editable text 비율
- table 구조 보존 여부
- row/cell 구조 보존 여부
- parent-child depth 유지율
- source_mapping 연결률
- stable identity 생성률
- empty node 비율
- high-risk node 비율

자동 Fail 후보 예시:

- page 매칭 실패
- editable text 비율이 지나치게 낮음
- table 문서인데 table/cell 구조가 거의 사라짐
- source_mapping 누락이 큼
- node hierarchy가 심하게 붕괴

### 8-2. 2단계: 사람 작업 가능성 검증

사람검증은 아래 3개 결과만 주면 충분하다.

- `usable`
- `usable_with_fix`
- `not_usable`

질문 예시:

- 이 문서를 Figma에서 바로 수정해도 되는가
- 표/셀/영역 수정이 가능한가
- 텍스트 수정과 이동이 자연스러운가
- 레이어 구조가 이해 가능한가
- 일부 보정만 하면 계속 사용할 수 있는가

최소 적용 범위:

- 사람검수는 우선 슬라이드 `12`, `26`, `29`, `34`에 대해 반드시 수행한다.

### 8-3. 3단계: Figma 변환 검증

확인할 것:

- 페이지 수 매칭
- frame/group/node 계층
- editable text 여부
- layout 붕괴 여부
- 표/셀 유지 여부
- asset 분리 여부

결과물:

- 시각 검증 리포트
- 구조 검증 리포트
- 수동 보정 필요 항목

### 8-4. 4단계: 운영 적합성 검증

확인할 것:

- 특정 node에 annotation 연결 가능한가
- 특정 셀/영역에 ownership 연결 가능한가
- source_mapping으로 원본 추적 가능한가
- node/page/document 단위 검색 projection이 가능한가
- 이후 diff 대상이 되는 stable key를 만들 수 있는가

결과물:

- 운영 활용성 체크리스트
- sync/search 재사용 가능 여부 판정

## 9. 자동검증 설계 초안

### 9-1. 자동검증 점수 구조

자동검증은 총점 하나보다 아래 3개 점수로 나누는 것이 좋다.

- `visual_proxy_score`
- `structure_score`
- `operability_score`

이렇게 해야 시각은 괜찮지만 구조가 약한 문서를 분리할 수 있다.

### 9-2. 자동검증 입력 데이터

- PPT parser output
- generated canonical graph
- generated Figma metadata
- source_mapping 결과
- node hierarchy 결과

### 9-3. 자동검증 결과 용도

- PoC 엔진 버전 비교
- 고위험 문서 자동 선별
- 수동 검수 우선순위 선정
- 운영 투입 가능 여부 1차 판정

## 10. 사람검증 설계 초안

### 10-1. 사람검증의 목적

사람검증은 "업무 지속 가능성" 판정이다.

즉 아래 질문에 답하면 된다.

"이 Figma 결과물로 실제 기획 작업을 계속할 때 무리가 없는가?"

### 10-2. 사람검증 메모 항목

점수보다 아래 메모가 중요하다.

- 작업이 막히는 이유
- 가장 불편한 구조
- 보정이 필요한 대표 유형
- 반복적으로 발생하는 변환 실패 패턴

## 11. 검증 체크리스트

### 11-1. parser 체크리스트

- slide 누락 없음
- text 누락률 허용 범위 내
- table 검출 가능
- row/cell 검출 가능
- group 후보 식별 가능
- source path 저장 가능

### 11-2. Figma 결과 체크리스트

- text editable
- frame/group 구조 존재
- table가 단순 이미지로 깨지지 않음
- 영역/섹션 분리 가능
- asset가 별도 분리됨
- page order 유지

### 11-3. 운영 적합성 체크리스트

- annotation 부착 가능
- ownership 연결 가능
- policy link 가능
- search indexing 가능
- sync diff 가능
- remap 가능

## 12. 계량 지표 초안

정확한 수치는 추후 PoC 결과를 보고 보정해야 하지만, 우선 아래 지표를 관리하는 것이 좋다.

- 페이지 매칭률
- 텍스트 추출 성공률
- editable text 비율
- table 구조 보존율
- row/cell 구조 보존율
- 계층 보존율
- source_mapping 연결률
- 운영 연결 가능 node 비율
- 수동 보정 필요 node 비율
- high-risk 문서 비율

## 13. PoC 단계에서 반드시 확인할 것

PoC는 기능 시연보다 품질 실패 지점을 찾는 데 목적이 있다.

반드시 확인할 항목:

- PPT 유형별 실패 패턴
- 표/셀 처리 한계
- 수동 배치된 요소의 해석 한계
- Figma node identity 생성 방식
- 이후 sync/search 연결이 어려운 node 유형
- 어떤 문서는 자동 변환보다 반자동 보정이 현실적인지

## 14. 권장 운영 방식

### 14-1. 품질 검증 보드 운영

샘플 문서별로 아래를 기록한다.

- 문서 유형
- 난이도
- 자동검증 결과
- 사람검증 결과
- 구조 품질 등급
- 운영 적합성 등급
- 주요 실패 원인
- 수동 보정 필요 여부

### 14-2. 승인 기준 운영

초기에는 모든 문서를 같은 기준으로 처리하지 않는다.

권장 구분:

- 자동 승인 가능 문서
- 검수 후 승인 문서
- 반자동 보정 필요 문서
- 자동 변환 제외 문서

### 14-3. 현실적 판단 기준

모든 PPT를 완벽하게 변환하는 것을 목표로 두면 일정이 무너진다.

따라서 품질 검증의 진짜 목적은 다음을 구분하는 것이다.

- 자동 변환에 적합한 문서 유형
- 일부 보정 후 운영 가능한 문서 유형
- 구조적 한계로 별도 처리해야 하는 문서 유형

## 15. 최종 품질 판정 원칙

최종 품질 판정은 아래 기준으로 내린다.

1. 자동검증에서 구조적 최소 기준을 통과했는가
2. 사람검증에서 "Figma로 작업을 이어갈 수 있다"는 판단이 나왔는가
3. 운영 적합성 관점에서 sync, search, annotation, ownership 연결이 가능한가

이 셋을 함께 만족해야 실제 합격이다.

## 16. 결론

이 프로젝트의 변환 품질 검증은 단순 비교 테스트가 아니다.

검증 질문은 최종적으로 아래 한 문장으로 정리할 수 있다.

"이 PPT를 Figma로 옮긴 결과가, 이후 운영 포털과 sync와 검색과 담당 연결의 기반 데이터로 계속 살아남을 수 있는가?"

이 질문에 예라고 답할 수 있어야 고품질 변환이다.
