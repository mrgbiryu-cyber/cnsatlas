# AX 전환 프로젝트 상사 보고용 Deck Plan

## 1. 보고 목적

이 보고의 목적은 단순히 "PPT를 Figma로 옮기는 도구를 만들고 있다"를 설명하는 것이 아니다.

이 보고는 아래 3가지를 승인받기 위한 자료다.

1. 왜 이 프로젝트가 필요한가
2. 이 프로젝트가 조직 생산성과 기획 운영 방식에 어떤 변화를 만드는가
3. 어떤 단계로 개발하고, 어떤 결과물을 기대할 수 있는가

즉, `기획 산출물 운영 방식의 전환`을 설명하는 보고로 본다.

---

## 2. 보고의 핵심 메시지

이 보고는 아래 한 문장으로 요약되어야 한다.

> 기존 PPT 중심 기획 업무를 Figma 기반 운영 체계로 전환하고, 전환된 결과를 구조화된 데이터로 적재해 정책 탐색과 질의응답까지 가능한 기획 운영 플랫폼으로 확장한다.

상사 관점에서는 아래 4개가 분명해야 한다.

- 왜 필요한가
- 무엇이 달라지는가
- 어디까지 가능해지는가
- 어떻게 단계적으로 추진하는가

---

## 3. 보고 흐름

### Slide 1. 제목 / 프로젝트 한 줄 정의

핵심 메시지:

- `PPT 기반 기획 산출물의 Figma 전환 및 운영 플랫폼화`

슬라이드 목적:

- 프로젝트를 단순 변환 자동화가 아니라 `업무 체계 전환`으로 보이게 만든다.

권장 화면:

- 왼쪽: 기존 PPT 문서 묶음 이미지
- 오른쪽: Figma + 내부 포털 + 검색 결과가 연결된 개념 이미지

---

### Slide 2. 왜 지금 필요한가

핵심 메시지:

- 현재 기획 산출물은 PPT, Figma, 정책 문서, 담당 정보가 분산되어 있어 운영 비용이 크다.

넣을 내용:

- 최신본 판단 어려움
- 정책/설명/담당 정보 분산
- 특정 화면 의도 파악 시간 과다
- 신규 인력 온보딩 어려움
- 검색과 재사용 어려움

권장 화면:

- "현재 상태" 문제를 보여주는 분산된 문서/채널/메신저/파일 구조 이미지

---

### Slide 3. 우리가 해결하려는 문제

핵심 메시지:

- 이 프로젝트는 단순 변환기가 아니라 `산출물 운영 + 지식 연결 + 검색` 문제를 동시에 푼다.

넣을 내용:

- PPT 초기 이관
- Figma 중심 운영
- 정책/담당/설명 연결
- 검색 및 질의응답
- 향후 변경 이력 및 sync 기반 운영

권장 화면:

- PPT -> Figma -> Node DB -> 정책/검색/QA 로 이어지는 흐름도

---

### Slide 4. 목표 상태

핵심 메시지:

- 문서는 더 이상 정적 파일이 아니라, 운영 가능한 자산이 된다.

넣을 내용:

- 화면별 정책 연결
- 영역별 담당자 확인
- 문서/페이지/노드 단위 탐색
- 관련 문서와 근거 기반 검색
- 질의응답 가능

권장 화면:

- 하나의 포털 화면 안에서 문서, 정책, 담당, 검색 결과가 함께 보이는 미래 상태 이미지

---

### Slide 5. 핵심 기능 1: PPT -> Figma 고품질 전환

핵심 메시지:

- 기존 PPT 사용자가 Figma로 넘어와도 바로 이해하고 이어서 작업할 수 있어야 한다.

넣을 내용:

- read-first 품질 원칙
- 텍스트 / 표 / 박스 / 흐름 구조 보존
- 단순 이미지화가 아닌 구조적 전환

권장 화면:

- Before/After 비교
  - 원본 PPT 일부
  - 변환된 Figma 일부

주의:

- 이 슬라이드는 AI 생성 이미지보다 실제 샘플 캡처가 더 설득력 있다.

---

### Slide 6. 핵심 기능 2: Node 구조화

핵심 메시지:

- 전환 결과는 단순 화면이 아니라 page / node / cell 단위로 구조화된다.

넣을 내용:

- document
- page
- node
- table / row / cell
- source mapping

권장 화면:

- 하나의 화면이 여러 노드로 분해되는 구조도
- 예: 화면 -> 박스 -> 셀 -> 텍스트

---

### Slide 7. 핵심 기능 3: 정책 탐색 / 질의응답

핵심 메시지:

- 전환된 결과는 단지 보는 용도가 아니라, 정책 찾기와 질의응답에 활용된다.

넣을 내용:

- "이 영역은 누가 담당인가?"
- "이 셀의 기획 의도는 어디 적혀 있나?"
- "이 화면과 연결된 정책 문서는 무엇인가?"

권장 화면:

- 내부 포털 검색 화면
- 질문 입력 -> 결과 -> 관련 문서/노드/정책이 같이 나오는 화면

---

### Slide 8. 기대 효과

핵심 메시지:

- 이 프로젝트는 기획 생산성과 운영 효율을 동시에 높인다.

넣을 내용:

- 문서 탐색 시간 감소
- 담당자 확인 시간 감소
- 정책 연결 정확도 향상
- 재사용률 증가
- 온보딩 속도 개선

권장 화면:

- KPI 카드 형태
- 검색 시간 단축 / 재사용률 증가 / 온보딩 기간 단축 등을 카드형으로 표현

---

### Slide 9. 추진 방식

핵심 메시지:

- 무리하게 전체를 한 번에 만드는 것이 아니라 단계별 검증으로 추진한다.

넣을 내용:

- 1단계: PPT -> Figma 품질
- 2단계: node 구조화
- 3단계: DB 적재 및 정책 탐색
- 4단계: 질의응답 / 포털 고도화

권장 화면:

- 4단계 로드맵 타임라인

---

### Slide 10. 현재 상태와 다음 액션

핵심 메시지:

- 현재는 Figma 고품질 변환 품질을 올리는 단계이고, 이후 node 적재와 검색으로 확장한다.

넣을 내용:

- 현재 benchmark 기반 품질 검증 진행
- read-first 품질 기준 확정
- 1/2페이지 우선 PASS 목표
- 이후 DB / 정책 탐색 단계 진행

권장 화면:

- 현재 상태 / 다음 단계 / 기대 결과 3단 구성

## 6. 이미지 생성 시 주의

AI 이미지 생성 시에는 아래를 지켜야 한다.

1. 텍스트가 많은 UI는 실제 문구를 넣으려 하지 않는다.
- AI는 UI 텍스트를 자주 깨뜨린다.
- 제목이나 라벨은 최소화한다.

2. 설명용 이미지는 `구조와 분위기` 중심으로 만든다.
- 대시보드
- 연결 구조
- 검색 결과 레이아웃
- 문서/노드/정책 관계

3. 실제 기능 시연은 AI 이미지로 대체하지 않는다.
- 특히 PPT -> Figma 전환 품질은 실제 결과 캡처가 중요하다.

## 10. 바로 쓸 수 있는 보고용 문장

### 프로젝트 정의

> 본 프로젝트는 기존 PPT 기반 기획 산출물을 Figma 중심 운영 체계로 전환하고, 전환 결과를 구조화된 데이터로 적재해 정책 탐색과 질의응답까지 가능한 기획 운영 플랫폼으로 확장하는 것을 목표로 합니다.

### 기대 효과

> 이 프로젝트를 통해 기획 산출물은 단순 파일이 아니라 정책, 담당, 설명, 이력이 연결된 운영 자산으로 전환되며, 문서 탐색 시간 단축과 재사용률 향상, 검수 효율 개선이 기대됩니다.

### 현재 단계

> 현재는 PPT -> Figma 고품질 전환 품질을 우선적으로 검증하고 있으며, 이후 node 기반 데이터 적재와 정책 탐색 기능으로 확장할 계획입니다.
